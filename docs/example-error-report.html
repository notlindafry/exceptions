Risk Ledger — validation error example

This page documents what risk-ledger does when the risk register contains invalid data. It was produced from a deliberately broken copy of the demo corpus and is kept here as a reference — the committed corpus is valid. To reproduce: set RISK-ACCT-TAKEOVER's probability_of_realization_90ci to [0.005, 1.5] in data/risks.yaml and run risk-ledger validate.

What was broken

A single field in data/risks.yaml, on the RISK-ACCT-TAKEOVER baseline:

  • before (valid): probability_of_realization_90ci: [0.005, 0.02]
  • after (invalid): probability_of_realization_90ci: [0.005, 1.5] — a probability is a number in (0, 1), so 1.5 cannot be the high end of a 90% CI.

The honesty gate for a malformed baseline is an error, not a flag: the risk can no longer anchor any estimate, so the risk and every exception mapped to it are rejected, and the CLI exits non-zero. A malformed record cannot be merged silently.

What risk-ledger report does

Unlike validate, the report command does not abort on a malformed risk — it excludes the rejected records and recomputes. The page still renders and the process exits 0, but the broken risk and its 13 exceptions silently disappear: the header drops from 14 mapped risks to 13, and the Data confidence section gains the line "13 records failed a hard gate and were rejected outright." This is exactly why validate is the gate to run first — it is wired to exit non-zero so CI can block the merge before any report built on the broken corpus is trusted.
